# 新加坡小学数学辅导 App 需求文档（2026-03 审核版）

## 1. 文档目标与范围

### 1.1 目标
定义当前项目的可执行需求基线，确保需求、架构、API 与实施计划一致，避免“文档说已支持但代码未落地”的情况。

### 1.2 产品范围
- 面向 P1-P6 学生家长的数学辅导 Web 应用。
- 当前核心能力是文字题输入后的 AI 解题与讲解。
- 题目来源以家长输入和内置题库为主，OCR 与拍照识题暂不在当前交付范围。

### 1.3 用户角色
- 家长：注册登录、管理学生、发起解题、查看历史与知识点练习趋势。
- 学生：通过家长账号下的学生档案间接使用。

---

## 2. 业务原则（需求评审后确认）

- 先保证“解题闭环可靠”，再做“学习路径智能化”。
- 所有“计划中”能力必须显式标注，不与“已上线能力”混写。
- 以可验证指标替代口号指标。例如：当前不承诺“首字 < 500ms SSE 真流式”，因为现实现是分段返回而非 token 级流式。

---

## 3. 功能需求

### 3.1 已实现（MVP 基线）

### FR-1 账号与认证
- 支持家长注册、登录并获取 JWT。
- 除 `/api/v1/auth/**` 与 `/actuator/**` 外，其余接口默认需要鉴权。

验收标准：
- 未携带 Token 调用受保护接口返回 401。
- 有效 Token 可正常访问 `solve`、`students`、`records`、`knowledge`。

### FR-2 解题能力
- 输入：`question` + `grade`（可选 `studentId`）。
- 流程：RAG 检索 -> Planner -> Content，输出 `parentGuide`、`childScript`、`barModelJson`、`knowledgeTags`。
- 提供 JSON 接口和 SSE 分段接口。

验收标准：
- `POST /api/v1/solve` 返回结构化 JSON。
- `POST /api/v1/solve/stream` 返回分段事件与 `[DONE]` 结束标记。

### FR-3 学生档案
- 支持在当前家长账户下创建与查询学生。

验收标准：
- `POST /api/v1/students` 返回学生 `id`、`name`、`grade`。
- `GET /api/v1/students` 返回该家长名下学生列表。

### FR-4 记录与知识点追踪
- 传入 `studentId` 解题时，持久化解题记录到 `solve_records`。
- 同时按 `knowledgeTags` 更新 `knowledge_progress.attempt_count`。

验收标准：
- `GET /api/v1/records/{studentId}` 可分页查询记录。
- `GET /api/v1/knowledge/{studentId}` 返回按练习次数排序的知识点追踪数据。

### FR-5 登录态持久化
- 页面刷新后无需重新登录（Token 本地持久化 + 过期处理）。
- `localStorage` 存储 JWT 与 `expiresAt`，`App()` 启动时恢复会话。
- Ktor 客户端 401 拦截器自动清除过期 Token 并跳转登录页。

验收标准：
- 登录后刷新页面直接进入解题页，无需再次输入密码。
- Token 过期后自动退出。

### FR-6 学生管理重设计
- `StudentManagementDialog` 弹窗：学生列表 + 年级选择器（P1–P6）+ 删除按钮。
- `DELETE /api/v1/students/{id}` 校验归属，级联删除关联记录。

验收标准：
- 新增学生可选 P1–P6 年级。
- 删除学生后列表刷新，关联的解题记录和知识点进度同步删除。

### FR-7 知识图谱与评估题库
- `knowledge_nodes` 表 63 个节点（P1–P6 全覆盖），树形结构展示。
- 掌握度三级：`UNKNOWN` / `FAMILIAR` / `MASTERED`，支持家长手动标记和评分自动建议。
- 题库 68 道按知识点标注的测评题，支持按知识点/年级随机抽题。
- 解题历史星级评分（1–5 星），评分自动更新关联知识点掌握度。

验收标准：
- `GET /api/v1/knowledge/graph` 返回完整知识图谱树。
- 知识图谱页面可展开/收起节点，点击切换掌握度。
- 历史页支持星级评分，评分后知识点掌握度自动更新。

### 3.2 暂不纳入当前基线

- OCR 拍照识题。
- 自适应学习路径与自动推题。
- 生产多云部署方案扩展。

## 4. 非功能需求

### NFR-1 性能与体验
- 冷启动首题允许秒级到十几秒级（受本地 LLM 性能影响）。
- 相同题目命中缓存时，目标响应 < 200ms（本地开发环境）。
- SSE 当前为“分段输出”而非 token 级流式，前后端文档需一致说明。

### NFR-2 可靠性
- LLM 调用需有超时保护（当前目标 60s）与明确错误码。
- 异常响应优先统一为 `{code, message}`；对历史接口中 `{error: ...}` 的返回作为技术债记录并收敛。

### NFR-3 数据与安全
- JWT 无状态鉴权。
- 仅最小化存储教学所需数据（用户、学生、记录、知识点追踪）。
- 生产环境需通过环境变量注入密钥，不写入仓库。

### NFR-4 可维护性
- 数据库结构由 Flyway 管理。
- 需求文档与 API 文档必须可追溯到实际接口字段。

## 5. 技术约束

- 后端：Java 25 + Spring Boot 4.0.3 + Spring AI 2.0.0-M2。
- 前端：KMP + Compose Wasm。
- 数据层：PostgreSQL 17（pgvector）+ Redis 7。
- 本地推理：Ollama（`qwen3.5:2b` + `nomic-embed-text`）。
- 生产主目标推理服务：DeepSeek（OpenAI 兼容方式）。

## 7. 本次审核清理结果

- 清理了过期的“4 周冲刺计划”，改为与当前 phase 制一致的需求基线。
- 移除了当前无法保证的“首字延迟 < 500ms”承诺。

- 强化了“已实现 / 进行中 / 暂不纳入”边界，避免需求范围漂移。
